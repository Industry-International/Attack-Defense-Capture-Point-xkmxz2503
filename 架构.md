# 攻防战占领点 Mod 架构文档

## 一、项目概述

一个 Minecraft NeoForge 1.21.1 模组，提供攻防战中的据点和区域管理功能。
玩家可以通过**据点方块**、**管理器物品**、**节点图 GUI** 和 **命令** 四种方式管理
据点和区域的占领关系。支持**多队伍自动占领**（基于 Scoreboard 团队）、**占领进度条 HUD**。

**核心概念**：
- **据点** = (中心坐标, 半径) 定义的世界区域，方块仅为放置标记
- **区域** = 据点的逻辑集合，用于组织攻防关系
- **两套独立的区域接口**：依赖接口（声明结构关系） + 解锁接口（控制运行时访问）

## 四、核心架构 — 分层数据访问

```
                         ┌─────────────────────────────────────┐
                         │       消费者层                       │
                         │  CapturePointBlockEntity             │
                         │  ModCommands / CapturePointGUI       │
                         └──────────┬──────────────────────────┘
                                    │ 依赖
                         ┌──────────▼──────────────────────────┐
                         │     ICaptureDataAccess 接口          │  ← 统一契约
                         │  (read / write / query / batch)      │
                         └──────────┬──────────────────────────┘
                                    │ implements
                         ┌──────────▼──────────────────────────┐
                         │   ServerCaptureDataAccess            │  ← 服务端实现
                         │   (单纯委派CaptureManager)           │
                         └──────────┬──────────────────────────┘
                                    │ delegates
                         ┌──────────▼──────────────────────────┐
                         │   CaptureManager (SavedData)         │  ← 数据持久化层
                         │   points  : Map<String, PointEntry>  │
                         │   zones   : Map<String, ZoneEntry>   │
                         │   layouts : Map<String, NodeLayout>  │
                         │   decisions: Map<String, DecisionData>│
                         └─────────────────────────────────────┘
```

### 设计要点

- **`ICaptureDataAccess`** 定义所有数据操作的统一契约，包含据点CRUD、区域CRUD、查询方法、批量快照、版本控制、队伍/进度操作、**解锁依赖操作**
- **`ServerCaptureDataAccess`** 包装 `CaptureManager`，所有操作单纯委派给底层 SavedData
- 消费者不直接依赖 `CaptureManager`，只依赖接口，便于测试和未来替换
- 获取方式：`ICaptureDataAccess.server(serverLevel)`

## 五、数据模型

### CapturePointEntry (据点记录)

```java
record CapturePointEntry(
    String name,                    // 唯一名称
    BlockPos pos,                   // 世界坐标（区域中心）
    boolean captured,               // 是否已被占领
    @Nullable String ownerTeam,     // 占领团队 (null=无主)
    @Nullable String capturingTeam, // 正在占领中的团队 (null=无)
    int captureProgress,            // 占领进度 0-100 (100=已完成)
    @Nullable String lastOwnerTeam, // 上一任占领者 (用于自动恢复)
    double radius,                  // 判定半径 (1-100)
    int displayColor,               // 显示颜色 (ARGB)
    boolean showRange               // 是否显示范围轮廓
)
```

**关键常量**：`MAX_PROGRESS = 100`

### ZoneEntry (区域记录)

```java
record ZoneEntry(
    String name,                  // 唯一名称
    List<String> capturePoints,   // 包含的据点名称列表
    @Nullable String requiredZone,// 前置依赖区域 (zone_out→required_zone接口, 结构声明)
    boolean captured,             // 是否已被占领
    @Nullable String ownerTeam,   // 占领团队
    List<String> unlockDependencies // 解锁依赖 (unlock_out→unlock_in接口, 运行时控制)
)
```

**两种接口的职责分离**：

| 接口 | 端口 | 用途 | 运行时影响 |
|------|------|------|-----------|
| **依赖接口** | `zone_out` → `required_zone` | 区域间结构关系声明 | ❌ 不影响 `canAccessZone()` |
| **解锁接口** | `unlock_out` → `unlock_in` | 区域运行时解锁控制 | ✅ `canAccessZone()` 唯一判断依据 |

### DecisionNodeData (判断器节点持久化)

```java
record DecisionNodeData(
    String name,                // 节点名称
    float x, float y,           // 布局坐标
    String condition,           // 条件类型 (captured/not_captured/owner_team/...)
    @Nullable String targetTeam,// 目标队伍
    int progressThreshold       // 进度阈值
)
```

## 六、节点图系统 (LDLib2 nodegraphtookit)

### 信号类型

三种信号类型，构成完全独立的信号路由通道：

| 信号 | TypeHandle | 用途 |
|------|-----------|------|
| **POINT_SIGNAL** | `capture:point_signal` | 据点→区域的成员关系 |
| **ZONE_SIGNAL** | `capture:zone_signal` | 区域→区域的依赖声明 |
| **UNLOCK_SIGNAL** | `capture:unlock_signal` | 区域→区域的解锁控制 |

### 节点类型与端口

| 节点 | 颜色 | 端口 | 端口容量 |
|------|------|------|----------|
| **CapturePointNode** | 绿色 `0xFF4CAF50` | 输出: `point_signal` (POINT_SIGNAL) | SINGLE |
| **CaptureZoneNode** | 蓝色 `0xFF42A5F5` | 输入: `point_in` (POINT_SIGNAL) — 成员据点 | **MULTIPLE** |
| | | 输出: `zone_out` (ZONE_SIGNAL) — 依赖声明 | SINGLE |
| | | 输入: `required_zone` (ZONE_SIGNAL) — 接收依赖 | SINGLE |
| | | 输入: `unlock_in` (UNLOCK_SIGNAL) — 接收解锁 | **MULTIPLE** |
| **CaptureConditionNode** | 橙色 `0xFFFF9800` | 输入: `point_target` (POINT_SIGNAL) / `zone_target` (ZONE_SIGNAL) | SINGLE |
| | | 输出: `true_out` / `false_out` (BOOLEAN_SIGNAL) | SINGLE |
| **LogicGateNode** | 紫色 `0xFF9C27B0` | 输入: `in_a` / `in_b` (BOOLEAN_SIGNAL) | SINGLE |
| | | 输出: `result` (BOOLEAN_SIGNAL) | SINGLE |
| **CaptureActionNode** | 红色 `0xFFE53935` | 输入: `trigger` (BOOLEAN_SIGNAL) | SINGLE |
| **ConstantNode** | 灰蓝 `0xFF78909C` | 输出: `value` (BOOLEAN_SIGNAL) | SINGLE |

### 连线类型与保存规则

**所有数据只从连线(WireModel)解析，不依赖节点选项值。删除连线即删除关系。**

| 连线 | 含义 | 保存到 |
|------|------|--------|
| `point_signal` → `point_in` | 据点属于该区域 | ZoneEntry.capturePoints |
| `point_signal` → `target` | 据点进入判断器 | 判断器条件路由 |
| `true_out`/`false_out` → `point_in` | 判断器路由据点到区域 | ZoneEntry.capturePoints |
| **`zone_out` → `required_zone`** | **区域依赖声明（纯结构）** | ZoneEntry.requiredZone |
| **`unlock_out` → `unlock_in`** | **区域解锁关系** | ZoneEntry.unlockDependencies |
| `unlock_out` → `unlock_target` | 区域进入判断器(解锁) | 判断器条件路由 |
| `unlock_true_out`/`false_out` → `unlock_in` | 判断器路由解锁 | ZoneEntry.unlockDependencies |

### 判断器条件路由流程 (buildSnapshotFromGraph)

```
Phase 1: 收集所有节点 → 分类为 PointModel / ZoneModel / DecisionModel
Phase 2: 解析所有连线 → 7种连线规则
Phase 3: 构建据点数据 (从节点选项)
Phase 4: 据点→判断器条件路由 (condition评估 → true_out/false_out)
Phase 5: 构建区域数据 (直连据点 + 判断器路由据点 + 直连接线解锁依赖)
Phase 6: 🔓 判断器解锁信号条件路由 (unlock_target → unlock_true_out/false_out)
         合并判断器路由解锁依赖到 ZoneEntry.unlockDependencies
规则②:  根据子据点占领状态重新计算区域 captured
```

### 保存流程

```
用户点击 Save
  ↓
saveGraph()
  ↓
buildSnapshotFromGraph() → (newPoints, newZones) + 布局 + 判断器数据
  ↓
版本冲突检测 (editMode && snapshotVersion != mgr.getVersion())
  ↓
applyGraphSnapshotWithLayout(newPoints, newZones, layouts, decisions)
  → 原子写入 CaptureManager
  ↓
syncAllBoundBlocks(serverLevel) → 方块实体同步
  ↓
broadcastToAll (CaptureDataSyncPayload) → 客户端缓存更新
```

### 持久化关键点

1. **所有关系只从连线解析** — 删除连线即删除关系，不依赖节点选项值回退
2. **判断器节点数据独立保存** — `DecisionNodeData` 通过 `CaptureManager` 持久化
3. **布局数据保存** — 节点位置通过 `NodeLayout` 持久化
4. **判断器检测** — 通过 `ICustomNodeModel.getNode() instanceof CaptureDecisionNode`，**不使用** `getClass().getName()` 字符串匹配

## 七、区域解锁逻辑 (运行时)

```java
canAccessZone(zoneName):
  1. zone = getZone(zoneName)
  2. if unlockDependencies 非空:
       递归检查所有解锁依赖的 canAccessZone
       全部通过 → 返回 true
  3. 无解锁依赖 → 区域无解锁约束，默认 true
```

**要点**：
- **解锁依赖由逻辑组件（CaptureActionNode）在运行时通过 CaptureManager API 控制**
- `unlock_in` 端口保留在 ZoneNode 上，作为接收解锁信号的声明性接口
- 区域不再负责发出解锁信号（已移除 unlock_out 端口）
- 条件节点 + 逻辑门节点 + 动作节点构成完整的"条件评估→执行解锁"链路
- `requiredZone`（`zone_out → required_zone`）仅为结构声明，**不影响运行时解锁**

## 八、Edit Properties 对话框的同步规则

编辑属性对话框中填写 `required_zone` 时，同时同步设置 `unlockDependencies`：

```
对话框保存 required_zone = "ZoneA"
  ├─ manager.setZoneRequiredZone(zoneName, "ZoneA")      // 依赖声明
  └─ manager.setZoneUnlockDependencies(zoneName, ["ZoneA"]) // 解锁依赖同步
```

这样对话框中的"依赖区域"字段同时控制两个接口，保证向后兼容。

## 九、服务端占领逻辑 (ServerTickHandler)

```
ServerTickEvent.Post (每 tick)
  ├─ tickCounter % 10 == 0 → processCaptureLogic(level)
  └─ tickCounter % 40 == 0 → syncAllBoundBlocks (安全网)

processCaptureLogic(ServerLevel):
  for each entry in getPoints():
    if !entry.showRange() → continue
    检测 entry.pos() 半径 entry.radius() 内的所有玩家
    统计各 Scoreboard 团队人数，取人数最多的队伍
    检查 canAccessZone (通过 unlockDependencies)
    根据区域锁定状态和队伍关系处理占领进度
```

### 占领判定流程

```
玩家站在据点范围内
  ├─ 区域被锁定 (canAccessZone = false) → 强制中立
  ├─ 无队伍 → 忽略（必须加入 Scoreboard 团队）
  ├─ 已占领且是己方队伍 → 逐渐恢复进度到100
  ├─ 已占领且是敌方队伍 → 清除进度
  ├─ 正在被己方占领 → 继续推进进度
  ├─ 正在被敌方占领 → 需要范围内无敌人时转为自己占领
  └─ 空闲 → 开始占领，设置 capturingTeam
```

### 原则

- **客户端永不直接写 CaptureManager** — 所有写入通过网络包发送到服务端
- **读取** — 客户端通过 `ClientCaptureDataCache` 读取同步数据
- **渲染** — `CapturePointWorldRenderer` 从 `ClientCaptureDataCache` 读取据点在世界上画圆
- **HUD** — `CaptureProgressOverlay` 从 `ClientCaptureDataCache` 读取队伍/进度渲染进度条
- **方块菜单的区域选择** — 从 `ClientCaptureDataCache.getZoneNames()` 读取区域列表，**禁止在客户端调用 `getServerDataAccess()`**
- **同步网络包** — `CaptureDataSyncPayload` 服务端→客户端，版本去重（含据点数据 + 区域名称列表）

## 十一、网络通信

### BlockEntityActionPayload (客户端→服务端)

```java
record BlockEntityActionPayload(BlockPos blockPos, String action, String data)
```

**动作字符串常量**：`ACTION_CREATE_POINT_AT`、`ACTION_SET_RADIUS`、`ACTION_SET_COLOR`、`ACTION_TOGGLE_RANGE`、`ACTION_SET_CAPTURED`、`ACTION_ZONE_SET_CAPTURED`、`ACTION_ADD_TO_ZONE`、`ACTION_REMOVE_FROM_ZONE`、`ACTION_BLOCK_UNBIND`

支持动作：
| action | data | 说明 |
|--------|------|------|
| `create_point_at` | `name,x,y,z,radius` | 创建据点 + 绑定方块 |
| `set_radius` | `name,radius` | 设置半径 |
| `set_color` | `name,color` | 设置颜色 |
| `toggle_range` | `name,true/false` | 切换范围显示 |
| `set_captured` | `name,true/false` | 设置占领状态 |
| `zone_set_captured` | `zoneName,true/false` | 设置区域占领状态 |
| `add_to_zone` | `zoneName,pointName` | 据点加入区域 |
| `remove_from_zone` | `pointName` | 据点移出区域 |
| `block_unbind` | `pointName` | 方块解除据点绑定 |

### CaptureDataSyncPayload (服务端→客户端)

```java
record CaptureDataSyncPayload(
    Map<String, PointRenderData> points,
    List<String> zoneNames           // 区域名称列表（供方块菜单使用）
)

record PointRenderData(
    BlockPos pos,
    double radius,
    int displayColor,
    boolean showRange,
    boolean captured,
    @Nullable String ownerTeam,
    @Nullable String capturingTeam,
    int captureProgress
)
```

- `zoneNames` 字段在 v1.0 重构后加入，使客户端缓存区域名称列表
- 方块菜单中"绑定到区域"对话框从此字段读取可用区域，无需访问服务端

同步触发时机：
- **玩家加入世界** — `PlayerLoggedInEvent` 发送完整数据
- **GUI 保存** — `saveGraph()` 后 `syncAllBoundBlocks` + `broadcastToAll`
- **方块操作** — `handleOnServer` 末尾 `broadcastToAll`
- **周期安全网** — 每 40 tick，版本去重

## 十二、方块系统

### CapturePointBlock

- 继承 `BaseEntityBlock`，渲染模式 `RenderShape.MODEL`
- 右键触发 `CapturePointBlockEntity.openUI()`（客户端打开菜单）

### CapturePointBlockEntity

- **6 个功能按钮**：创建据点、绑定区域、查看状态、设定半径、切换范围、移除绑定
- **方块持久化**：仅 `boundPointName` — 绑定到 NBT `saveAdditional/loadAdditional`
- **渲染缓存**（不持久化）：`radius`, `displayColor`, `showRange` — 从 CaptureManager 同步
- **写入路径**：所有操作通过 `sendAction()` → `BlockEntityActionPayload` 网络包 → 服务端
- **没有乐观更新**：所有写入只发网络包，不直接修改本地缓存
- **区域选择对话框**：从 `ClientCaptureDataCache.getZoneNames()` 读取区域列表，**不再直接访问服务端数据**（v1.0 重构修复多人在线下崩溃问题）

## 十四、渲染系统

### CapturePointBlockEntityRenderer

- 留空渲染器 — 方块模型由 `RenderShape.MODEL` 处理

### CapturePointWorldRenderer

- 使用 `RenderLevelStageEvent.AFTER_BLOCK_ENTITIES` 事件钩子
- 从 `ClientCaptureDataCache` 读取据点数据
- **以据点的实际中心坐标绘制**，而非方块位置
- 32 段圆环 (水平) + 4 条垂直辅助标记
- 视锥体裁剪 — 仅渲染视野内的据点

### ClientCaptureDataCache

- 客户端单例缓存，线程安全（synchronized 读写）
- 数据来源：`CaptureDataSyncPayload.handleOnClient()`
- **缓存内容**：据点渲染数据（`Map<String, PointRenderData>`）+ 区域名称列表（`List<String>`）
- **区域名称用途**：方块菜单的"绑定到区域"对话框从此缓存读取可用区域列表，**不再直接访问服务端数据**

### CaptureProgressOverlay

- 监听 `RenderGuiEvent.Pre` 事件
- 从 `ClientCaptureDataCache` 读取队伍/进度数据
- 仅玩家在据点范围内时显示

## 十七、版本去重与性能

`CaptureDataSyncPayload.broadcastToAll()` 内置版本去重：

```java
long currentVersion = access.getVersion();
Long lastVersion = LAST_BROADCAST_VERSION.get(dimKey);
if (lastVersion != null && lastVersion == currentVersion) return;
```

- 数据未变更时 **不发送** 网络包
- 单据点 ≈ 100 字节，50 个据点 ≈ 5KB
- 占领进度 tick 处理间隔 10 tick (0.5 秒)

## 十九、配置系统

`Config.java` 提供可配置的占领相关参数，取代了模版生成的空配置：

| 配置项 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `captureIntervalTicks` | int | 10 | 占领进度检查间隔（tick） |
| `syncIntervalTicks` | int | 40 | 同步安全网间隔（tick） |
| `idleTimeoutTicks` | int | 600 | 空闲自动恢复超时（tick，约30秒） |
| `captureSpeed` | int | 2 | 占领/清除速度（每 tick 进度值） |
| `recoverySpeed` | int | 1 | 自动恢复速度（每 tick 进度值） |

配置值通过 `ModConfigEvent` 加载，供 `ServerTickHandler` 和服务端逻辑使用。
